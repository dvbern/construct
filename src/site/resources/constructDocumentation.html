Usage part 1: Object factory

The following descriptions explain how to use ObjectConstructor and its related classes.

The UML diagram below shows the core classes:


XMLObjectConstructor is an implementation of ObjectConstructor that uses xml-files as definitions of objects.

The following example shows the usage of the default XMLObjectConstructor. When using the default version, the xml-files with the object definitions must be located in the CLASSPATH of the application: 

    //create default XMLObjectConstructor
    XMLObjectConstructor constructor = new XMLObjectConstructor();

    try {
        //The object retrived by the following  method call is declared in the xml-file 'myObject.xml'
        //'false' indicates, that the instantiated object has to be retrieved from cache, if already instantiated
        //(if 'true' is passed, the xml-file is always parsed and a new instance is returned)
        Object obj = constructor.construct("myObject.xml", false);

        //in the following example a java.util.Map is returned
        java.util.Map myMap = (java.util.Map)constructor.construct("myMap.xml", false);
    } catch (ConstructionException ex) {
        //... handle exception
    }

As the example shows, the usage of the XMLObjectConstructor is very simple. It's method construct() returns an object, that has to be casted.

The method construct() throws a ConstructionException, if the definition in the indicated xml-file is not correct or the file can not be located. The stack trace of the exception may include the stack trace of nested exceptions.

As the XMLObjectConstructor has caching function, always the same instance of it must be used for retrieving the same objects.

Internally the XMLObjectConstructor uses a ParserFactory which optionally accepts a ResourceLocator.
The ResourceLocator is responsible for finding the xml-files that define the objects. By default (as shown in the example above), XMLObjectConstructor uses a ClassLoaderResourceLocator. In that case, the xml-files must be located in the CLASSPATH. (But a ClassLoaderResourceLocator can be created with a specific ClassLoader).

The other solution is the use of a FilePathResourceLocator which is able to find the resources (xml-files) in the specified "path". If you want to use a FilePathResourceLocator, you must pass an instance of it to the ParserFactory. The FilePathResourceLocator requires the definition of a "path" (representing the directory where the files are stored) and optionally of a long-value (resChangeCheckPeriod) that defines the interval in ms at which the files are being rechecked for changes.

The advantage of the FilePathResourceLocator is the automatic detection of changes in the xml-files. If the content of a xml-file changes, the parsed information is removed from the cache and reparsed at the next request.

Example: 

    //create a locator which looks for xml-files in the directory "/myResourcePath" and
    //checks every minute, whether the xml-files have been changed
    FilePathResourceLocator locator = new FilePathResourceLocator("/myResourcePath", 60000);
    XMLObjectConstructor constructor = new XMLObjectConstructor(new ParserFactory(locator));
    ....[retrieve objects and so on]...
    //if the constructor isn't used any more: stop the checker thread
    locator.stopResourceChecker();

Usage part 2: Object definitions in XML

There are to ways to define an object. The first is defining an object by declaring a constructor-call, the second is defining an object by declaring a program-flow.

construct

If an object has to be instantiated simply by calling a constructor, the declaration in the xml-file is fairly simple.

Let's assume we have the following JAVA class: 

    package xyz;

    public class Hero {
        private String name;

        public Hero(String name, int age) {
            this.name = name;
            ...
        }
        public String getName() {
            return name;
        }
        ...
    }

Then the following xml-file would define an instance of Hero: 

    <?xml version="1.0" encoding="UTF-8"?>
    <construct class="xyz.Hero">
        <string value="Johnny Weissmueller" />
        <int value="33" />
    </construct>

<construct> is the root-tag, representing a constructor of an object. All the constructor arguments must be placed as nested tags.

The following tags are allowed as nested "argument" tags (for details see Element descriptions):

"Primitive" types and java.lang.String:

  • <string value="..." />
  • <int value="..." />
  • <short value="..." />
  • <long value="..." />
  • <float value="..." />
  • <double value="..." />
  • <char value="..." />
  • <boolean value="..." />

Nested construct tags:

  • <construct class="...">...</construct>

Class definitions:

  • <class name="..." />

    Returns the class with the indicated name.

References to other xml-files:

  • <ref id="..." />

    If the name of another xml-file is set as value of id, the <ref> tag returns the object that is defined in that xml-file.

Arrays:

  • <array elementtype="...">

    Examples for array definitions:

    The definition 

        <array elementtype="int">
        <int value="1"/>
        <int value="2"/>
        <int value="3"/>
     </array>
    represents an int array with one dimension and a length of 3 (equivalent in JAVA for example: int[] myArray = {1, 2, 3})

    An array with two dimensions and String elements could be defined the following way: 

        <array elementtype="java.lang.String[]">
        <array elementtype="java.lang.String">
            <string value="a string"/>
            <string value="another string"/>
        </array>
        <array elementtype="java.lang.String">
            <string value="yet a string"/>
            <string value="yet another string"/>
        </array>
    </array>
    Arrays of any dimension and type can be defined that way. Of course the use of nested <construct> tags is also allowed.

Sometimes a definition has to be casted. The following tag exists for this purpose:

  • <cast class="...">...</cast>

    • Example:

    Let's assume we have the following class: 

        package xyz;

    public class MyContainer {
        private Object object;

        public MyContainer(Object obj) {
            object = obj;
        }
        public Object getObject() {
            return object;
        }

        ...
    }

    The object can be defined as follows: 

        <construct class="xyz.MyContainer">
        <cast class="java.lang.Object"><string value="my string" /></cast>
    </construct>
    Because the constructor of MyContainer requires an object of type java.lang.Object and we want to pass a java.lang.String, we have to cast the argument to java.lang.Object.

script

Sometimes calling a constructor is not enough for instantiating a concrete object. In that case, the following tag is used instead of <construct>...</construct>:

  • <script>...</script>

Within <script> tags the additional use of the following tags is possible:

  • <vardef name="..." >...</vardef>

    Stores an object with a variable name, i.e. stores the object, that is declared by the nested tag(s), with the indicated name attribute. All the tags described in the previous chapter plus <script> can be used as nested tags.
    Nested <script> tags have its own scope; therefore variables can be "overriden" in nested <script> areas without changing variable definitions from embedding <script> tags.

  • <var name="..." />

    Returns the object that is stored with the name in the actual <script> scope or, if not found, in a scope of embedding <script> tags.
    I.e. an object must have been defined by <vardef name="..." >...</vardef> with the same value of name before.

  • <invoke methodName="..." >...</invoke>

    Invokes the method with the name indicated by methodName on the object or class defined by the nested <target>...</target> tag with the parameters defined by the nested <parameters>...</parameters> tag and returns the return value of the method.
    The invoked method must be public.

  • <getfield name="..." >...</getfield>

    Returns the value stored in the field with the name indicated by name of the object or class defined by the nested <target>...</target> tag.
    The field must be public.

  • <setfield name="..." >...</setfield>

    Stores the value defined by the nested <value>...</value> tag in the field with the name indicated by name of the object or class defined by the nested <target>...</target> tag.
    The field must be public.

  • <return>...</return>

    Defines the object returned by a <script>...</script> definition. Can contain any tags described in this documentation.
    Should be at the end of a <script>...</script> definition.

    If there is no <return>...</return> tag in a <script>...</script> definition, the last defined object is returned.

Example:

Let's assume the following scenario:

We have a xml definition of a MyContainer instance in a file named "lastname.xml": 

    <?xml version="1.0" encoding="UTF-8"?>
    <construct class="xyz.MyContainer">
        <cast class="java.lang.Object"><string value="Weissmueller" /></cast>
    </construct>

and another definition of a MyContainer instance in a file named "firstname.xml": 

    <?xml version="1.0" encoding="UTF-8"?>
    <construct class="xyz.MyContainer">
        <cast class="java.lang.Object"><string value="Johnny" /></cast>
    </construct>

We can now create a definition of a Hero instance in the file "myHero.xml" using the defined objects: 

<?xml version="1.0" encoding="UTF-8"?>
<script>
    <vardef name="firstname">
        <cast class="java.lang.String">
            <invoke methodName="getObject">
                <target><ref id="firstname.xml" /></target>
            </invoke>
        </cast>
    </vardef>
    <vardef name="lastname">
        <cast class="java.lang.String">
            <invoke methodName="getObject">
                <target><ref id="lastname.xml" /></target>
            </invoke>
        </cast>
    </vardef>
    <vardef name="tmp">
        <invoke methodName="concat">
            <target><var name="firstname" /></target>
            <parameters><string value=" " /></parameters>
        </invoke>
    </vardef>
    <vardef name="fullname">
        <invoke methodName="concat">
            <target><var name="tmp" /></target>
            <parameters><var name="lastname" /></parameters>
        </invoke>
    </vardef>
    <return>
        <construct class="xyz.Hero">
            <var name="fullname" />
            <int value="33" />
        </construct>
    </return>
</script>

The following code would then print "Johnny Weissmueller" to the System.out (assuming that the 3 xml-files and the classes MyContainer and Hero are in the CLASSPATH): 

    XMLObjectConstructor constructor = new XMLObjectConstructor();
    Hero hero = (Hero)constructor.construct("myHero.xml", true);
    System.out.println(hero.getName());
